Mini Juego 2D en Java: Esquiva Cuadrados

Guía completa sobre la arquitectura, el flujo de ejecución, cada clase y método, el ciclo de renderizado, los tests y cómo ampliar el juego.

Arquitectura del proyecto

UI + entrada

App.java

Punto de entrada. Crea el JFrame, instancia GamePanel, captura eventos de teclado y ejecuta el bucle de pintado mediante un Timer de Swing.

Estado + motor

GameEngine.java

Almacena todo el estado mutable: posiciones de jugador y enemigo, velocidades, puntuación y el flag running. Expone tick() y resetGame().

Reglas puras

GameLogic.java

Clase final sin estado con funciones static deterministas: límites del jugador, rebote del enemigo, detección de colisión y velocidad aleatoria con signo.

Calidad

Tests

6 tests unitarios en AppTest sobre GameLogic y el constructor de GameEngine. 4 tests de integración en GameEngineIntegrationTest para el ciclo completo.

Principios de diseño aplicados

🔀

Separación de responsabilidades

UI, estado y reglas están en clases distintas. Cambiar el renderizador no toca la lógica de juego.

💉

Inyección de dependencias

Random y todos los parámetros numéricos se inyectan al constructor de GameEngine, lo que permite tests con semillas fijas y resultados reproducibles.

🔒

Copias defensivas

getPlayer() y getEnemy() devuelven new Rectangle(original), protegiendo el estado interno del motor frente a modificaciones externas.

✅

Funciones puras y testables

GameLogic no tiene campos de instancia; solo recibe datos y devuelve resultados. Cero efectos secundarios y cobertura de test máxima.

Clases y métodos en detalle

App.java — Punto de entrada y ventana Swing UI

App contiene el método main y la clase interna privada GamePanel que extiende JPanel. Toda la lógica de presentación vive aquí.

main(String[] args)

Envuelve la construcción de la UI en SwingUtilities.invokeLater para garantizar que se ejecute en el Event Dispatch Thread (EDT) de Swing, evitando condiciones de carrera.

SwingUtilities.invokeLater(() -> { JFrame frame = new JFrame("Mini Juego 2D - Esquiva Cuadrados"); frame.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE); frame.setResizable(false); GamePanel gamePanel = new GamePanel(); frame.setContentPane(gamePanel); frame.pack(); // ajusta al preferredSize (800x500) frame.setLocationRelativeTo(null); // centra la ventana en pantalla frame.setVisible(true); gamePanel.requestFocusInWindow(); // imprescindible para recibir teclas });
GamePanel() — Constructor

Configura el panel, crea el GameEngine con todas las constantes, arranca el Timer a 16 ms e instala un KeyAdapter para captura continua de teclas.

setPreferredSize(new Dimension(WIDTH, HEIGHT)); // 800x500 px setBackground(new Color(18, 24, 40)); setFocusable(true); this.engine = new GameEngine( WIDTH, HEIGHT, PLAYER_SIZE, ENEMY_SIZE, PLAYER_SPEED, ENEMY_MIN_SPEED, ENEMY_MAX_SPEED, ENEMY_HORIZONTAL_SPEED, new Random()); this.gameTimer = new Timer(16, event -> { updateGame(); // avanza el estado un tick repaint(); // solicita redibujado al EDT }); this.gameTimer.start();
updateGame()

Delega el avance del estado a engine.tick(). Detecta la transición running → false para hacer sonar el pitido del sistema en el momento exacto del game over.

private void updateGame() { boolean wasRunning = engine.isRunning(); engine.tick(leftPressed, rightPressed, upPressed, downPressed); if (wasRunning && !engine.isRunning()) { Toolkit.getDefaultToolkit().beep(); // pitido de sistema al morir } }
paintComponent(Graphics g)

Llamado automáticamente por Swing en cada repaint(). Orden de capas (painter's algorithm):

  1. super.paintComponent(g) — limpia el fondo con #12182A
  2. Rejilla decorativa: líneas cada 40 px en azul oscuro #28405E
  3. Jugador: fillRect verde #50DC82 en la posición de engine.getPlayer()
  4. Enemigo: fillRect rojo #FF5F56 en la posición de engine.getEnemy()
  5. HUD: "Puntos: N" en fuente BOLD 22 px, esquina superior izquierda
  6. Instrucciones: "Mover: WASD o flechas" en la parte inferior
  7. Overlay "GAME OVER" centrado en amarillo (solo si !engine.isRunning())
KeyAdapter — teclas reconocidas
  • ← / A → activa/desactiva leftPressed
  • → / D → activa/desactiva rightPressed
  • ↑ / W → activa/desactiva upPressed
  • ↓ / S → activa/desactiva downPressed
  • R → llama a engine.resetGame() solo si !isRunning()

El uso de flags booleanos separados en keyPressed / keyReleased permite movimiento diagonal suave (dos teclas a la vez).

GameEngine.java — Motor y estado mutable del juego Motor

Clase final de ámbito de paquete (sin public). No tiene ninguna referencia a Swing, por lo que es perfectamente testable sin entorno gráfico.

Constructor — validación exhaustiva

Antes de asignar cualquier campo, valida todos los parámetros. Lanza NullPointerException si random es null, e IllegalArgumentException para dimensiones ≤ 0, tamaños ≤ 0, velocidad ≤ 0 o rango inválido (minSpeed > maxSpeed).

this.player = new Rectangle(80, height / 2 - playerSize / 2, playerSize, playerSize); this.enemy = new Rectangle(width - 120, 80, enemySize, enemySize); this.enemySpeed = enemyMinSpeed + random.nextInt(enemyMaxSpeed - enemyMinSpeed + 1);
tick(left, right, up, down) — corazón del motor

Se ejecuta aproximadamente 62 veces por segundo. Orden de operaciones dentro de un tick:

  1. Si !running, retorna inmediatamente (estado completamente congelado).
  2. Aplica las teclas pulsadas: player.x ±= playerSpeed, player.y ±= playerSpeed.
  3. Llama a GameLogic.clampPlayerToBounds para mantener al jugador dentro del lienzo.
  4. Mueve el enemigo verticalmente: enemy.y += enemySpeed.
  5. Invierte enemySpeed si el enemigo toca el borde superior (y ≤ 0) o inferior (y+h ≥ H).
  6. Mueve el enemigo horizontalmente: enemy.x -= enemyHorizontalSpeed (siempre hacia la izquierda).
  7. Si el enemigo sale por la izquierda: incrementa score y llama a resetEnemyPosition().
  8. Verifica colisión con GameLogic.hasCollision(player, enemy); si hay solapamiento, running = false.
void tick(boolean left, boolean right, boolean up, boolean down) { if (!running) return; if (left) player.x -= playerSpeed; if (right) player.x += playerSpeed; if (up) player.y -= playerSpeed; if (down) player.y += playerSpeed; GameLogic.clampPlayerToBounds(player, width, height); enemy.y += enemySpeed; enemySpeed = GameLogic.updateVerticalEnemySpeed( enemy.y, enemy.height, height, enemySpeed); enemy.x -= enemyHorizontalSpeed; if (enemy.x + enemy.width < 0) { score++; resetEnemyPosition(); } if (GameLogic.hasCollision(player, enemy)) running = false; }
resetGame()

Restaura completamente el estado inicial: score=0, running=true, jugador centrado verticalmente a x=80, enemigo en x=width−120, y=80 y velocidad vertical aleatoria con signo.

void resetGame() { score = 0; running = true; player.x = 80; player.y = height / 2 - player.height / 2; enemy.x = width - 120; enemy.y = 80; enemySpeed = GameLogic.randomSignedSpeed(random, enemyMinSpeed, enemyMaxSpeed); }
resetEnemyPosition() — reaparición del enemigo

Se llama cuando el enemigo cruza el borde izquierdo. La nueva posición es aleatoria: x entre 800 y 1020, y entre 0 y (H − enemySize), con velocidad aleatoria con signo.

enemy.x = width + random.nextInt(220); // 800–1020 enemy.y = random.nextInt(height - enemySize); // 0–468 enemySpeed = GameLogic.randomSignedSpeed(random, enemyMinSpeed, enemyMaxSpeed);
API pública
  • getPlayer() → copia defensiva del Rectangle del jugador
  • getEnemy() → copia defensiva del Rectangle del enemigo
  • getScore() → puntuación actual como int
  • isRunning() → true mientras el juego esté activo
  • setEnemyPosition(x, y) → usado exclusivamente en tests de integración
GameLogic.java — Funciones puras sin estado Lógica pura

Clase final con constructor privado. No tiene ningún campo de instancia. Todas las funciones son static, deterministas y sin efectos secundarios (salvo clampPlayerToBounds, que muta el Rectangle recibido por diseño explícito).

clampPlayerToBounds(Rectangle player, int width, int height)

Limita las coordenadas del jugador para que el rectángulo completo permanezca dentro del área de juego. Modifica el Rectangle directamente (no crea uno nuevo).

if (player.x < 0) player.x = 0; if (player.y < 0) player.y = 0; if (player.x + player.width > width) player.x = width - player.width; if (player.y + player.height > height) player.y = height - player.height;
updateVerticalEnemySpeed(int enemyY, int enemyH, int panelH, int currentSpeed)

Devuelve -currentSpeed cuando el enemigo toca el borde superior (enemyY ≤ 0) o el inferior (enemyY + enemyH ≥ panelH); en cualquier otro caso devuelve currentSpeed sin cambios. Función pura: no muta ningún parámetro.

if (enemyY <= 0 || enemyY + enemyHeight >= panelHeight) return -currentSpeed; return currentSpeed;
randomSignedSpeed(Random random, int min, int max)

Genera un entero en [min, max] y le asigna signo positivo o negativo de forma aleatoria. Produce movimiento vertical inicial hacia arriba o hacia abajo con la misma probabilidad.

int baseSpeed = min + random.nextInt(max - min + 1); // valor absoluto en [min,max] return random.nextBoolean() ? baseSpeed : -baseSpeed; // signo aleatorio
hasCollision(Rectangle player, Rectangle enemy)

Delega directamente en Rectangle.intersects() de Java AWT. Dos rectángulos colisionan si se solapan en ambos ejes; tocarse en un solo borde no cuenta como colisión.

return player.intersects(enemy); // true si los rectángulos se solapan en X e Y

Flujo completo de ejecución

  1. JVM arranca — App.main()

    SwingUtilities.invokeLater encola la construcción de la UI en el EDT de Swing. Esto garantiza thread-safety para todos los componentes gráficos.

  2. Construcción de GamePanel

    Se crea un GameEngine con las constantes del panel, se arranca el Timer(16 ms) y se registra el KeyAdapter. JFrame.pack() ajusta la ventana al preferredSize de 800×500.

  3. Bucle de juego — Timer cada 16 ms

    Cada disparo del Timer llama a updateGame() y luego a repaint(), produciendo aproximadamente 62 fotogramas por segundo.

    new Timer(16, event -> { updateGame(); // 1. avanza estado repaint(); // 2. solicita redibujado })
  4. engine.tick() — un frame de simulación

    Aplica la entrada del teclado, limita al jugador, mueve al enemigo (rebote vertical + avance horizontal), verifica salida por la izquierda (punto) y detecta colisión (game over).

  5. paintComponent(g) — renderizado de la escena

    Lee las posiciones con los getters defensivos de GameEngine y dibuja cada capa (fondo → rejilla → jugador → enemigo → HUD → overlay).

  6. Game Over detectado

    GameLogic.hasCollision devuelve true → running = false. El Timer sigue corriendo pero tick() retorna inmediatamente; solo se redibujan el overlay y el marcador.

  7. Reinicio con tecla R

    El KeyAdapter detecta R, comprueba que !engine.isRunning() y llama a engine.resetGame(). El siguiente tick reanuda el juego con el estado inicial.

Entrada de teclado con flags booleanos KeyAdapter

Se mantienen cuatro flags que se activan en keyPressed y se desactivan en keyReleased. El motor los lee en cada tick, permitiendo movimiento continuo y diagonal mientras las teclas estén pulsadas.

// En keyPressed: case KeyEvent.VK_LEFT: case KeyEvent.VK_A: leftPressed = true; break; case KeyEvent.VK_RIGHT: case KeyEvent.VK_D: rightPressed = true; break; case KeyEvent.VK_UP: case KeyEvent.VK_W: upPressed = true; break; case KeyEvent.VK_DOWN: case KeyEvent.VK_S: downPressed = true; break; case KeyEvent.VK_R: if (!engine.isRunning()) engine.resetGame(); break; // En keyReleased: los mismos casos con = false
Movimiento del enemigo: rebote vertical + avance horizontal tick

El enemigo combina dos movimientos independientes por frame:

  • Vertical (rebote): enemy.y += enemySpeed cada tick. Al tocar el borde superior o inferior, GameLogic.updateVerticalEnemySpeed invierte el signo.
  • Horizontal (avance): enemy.x -= 4 siempre. Cuando enemy.x + enemy.width < 0, el score sube 1 y el enemigo reaparece a la derecha con posición y velocidad aleatorias.
// Vertical (rebote en bordes) enemy.y += enemySpeed; enemySpeed = GameLogic.updateVerticalEnemySpeed( enemy.y, enemy.height, height, enemySpeed); // Horizontal (avance constante hacia la izquierda) enemy.x -= enemyHorizontalSpeed; // 4 px/frame if (enemy.x + enemy.width < 0) { score++; resetEnemyPosition(); // x > 800, y y velocidad aleatorios }
Pipeline de renderizado — capas en orden paintComponent

Swing llama a paintComponent cuando hay un repaint pendiente. Se dibuja de atrás hacia adelante (painter's algorithm):

  1. super.paintComponent(g) — rellena el fondo con #12182A
  2. Rejilla: g.drawLine cada 40 px en azul oscuro (new Color(40, 60, 94))
  3. Jugador verde: fillRect(player.x, player.y, …) con new Color(80, 220, 130)
  4. Enemigo rojo: fillRect(enemy.x, enemy.y, …) con new Color(255, 95, 86)
  5. HUD de puntuación: drawString("Puntos: " + score, 20, 35)
  6. Instrucciones en el margen inferior
  7. Overlay de Game Over solo si !engine.isRunning()

Tests — cobertura completa

AppTest — Tests unitarios sobre GameLogic y constructor 6 tests

Cada test verifica una función pura de GameLogic o la validación de argumentos del constructor de GameEngine. No requieren entorno gráfico.

clampPlayerToBounds_keepsPlayerInsideBoard

Jugador en posición fuera del área (x=−10, y=900). Tras el clamp en un panel 800×500 con tamaño 36, debe quedar en (0, 464) ya que 500−36=464.

Rectangle player = new Rectangle(-10, 900, 36, 36); GameLogic.clampPlayerToBounds(player, 800, 500); assertEquals(0, player.x); assertEquals(464, player.y);
updateVerticalEnemySpeed_invertsOnTopAndBottomBorders

Verifica los tres casos posibles: borde superior (y=0 → invierte de 5 a −5), borde inferior (y=468, h=32, panel=500 → invierte de −4 a 4) y posición media (sin cambio).

assertEquals(-5, GameLogic.updateVerticalEnemySpeed(0, 32, 500, 5)); assertEquals( 4, GameLogic.updateVerticalEnemySpeed(468, 32, 500, -4)); assertEquals( 3, GameLogic.updateVerticalEnemySpeed(120, 32, 500, 3));
hasCollision_returnsTrueOnlyWhenRectanglesOverlap

Un enemigo que se solapa con el jugador (120,120) → true. Uno que está lejos (400,300) → false.

Rectangle player = new Rectangle(100, 100, 36, 36); assertTrue( GameLogic.hasCollision(player, new Rectangle(120, 120, 32, 32))); assertFalse(GameLogic.hasCollision(player, new Rectangle(400, 300, 32, 32)));
randomSignedSpeed_staysInExpectedAbsoluteRange

Ejecuta 200 iteraciones con semilla fija (42). El valor absoluto de la velocidad siempre debe estar en el rango [3, 7], tanto positiva como negativa.

Random random = new Random(42); for (int i = 0; i < 200; i++) { int speed = GameLogic.randomSignedSpeed(random, 3, 7); assertTrue(Math.abs(speed) >= 3 && Math.abs(speed) <= 7); }
gameEngine_constructorRejectsInvalidSpeedRange

Pasar enemyMinSpeed=8 con enemyMaxSpeed=7 (min > max) debe lanzar IllegalArgumentException.

assertThrows(IllegalArgumentException.class, () -> new GameEngine(800, 500, 36, 32, 6, 8, 7, 4, new Random(1)));
gameEngine_constructorRejectsNullRandom

Pasar null como instancia de Random debe lanzar NullPointerException de forma explícita (validación con Objects.requireNonNull).

assertThrows(NullPointerException.class, () -> new GameEngine(800, 500, 36, 32, 6, 3, 7, 4, null));
GameEngineIntegrationTest — Ciclo completo del motor 4 tests

Usan un GameEngine con semilla fija new Random(123) para resultados reproducibles. Verifican comportamientos de varios ticks encadenados.

tick_movesEnemyLeftOnEachFrame

Con todas las teclas en false, el enemigo debe desplazarse exactamente 4 px a la izquierda por cada tick (ENEMY_HORIZONTAL_SPEED = 4).

int initialEnemyX = engine.getEnemy().x; engine.tick(false, false, false, false); assertEquals(initialEnemyX - 4, engine.getEnemy().x);
tick_whenEnemyLeavesScreen_addsPointAndRespawns

Se fuerza el enemigo en x = −enemyWidth. El siguiente tick debe sumar 1 punto y reubicar al enemigo a x ≥ 800, con el juego aún en marcha.

engine.setEnemyPosition(-enemy.width, 120); engine.tick(false, false, false, false); assertEquals(1, engine.getScore()); assertTrue(engine.getEnemy().x >= 800); assertTrue(engine.isRunning());
tick_whenCollisionHappens_stopsGameAndNextTicksDoNotAdvance

Se coloca el enemigo sobre el jugador para forzar colisión. Tras un tick el juego se detiene. Ticks adicionales (incluso con tecla izquierda) no modifican posición ni score.

engine.setEnemyPosition(player.x, player.y); // colisión directa engine.tick(false, false, false, false); assertFalse(engine.isRunning()); int frozenX = player.x; int frozenScore = engine.getScore(); engine.tick(true, false, false, false); // intento de movimiento assertEquals(frozenX, player.x); // x no cambia assertEquals(frozenScore, engine.getScore()); // score no cambia
getPlayer_returnsDefensiveCopy

Modificar el Rectangle devuelto por getPlayer() no debe alterar la posición interna del motor, que debe seguir siendo x=80.

Rectangle snapshot = engine.getPlayer(); snapshot.x = 999; // modifica la copia assertEquals(80, engine.getPlayer().x); // estado interno intacto
Ejecutar los tests con Maven CLI
mvn test # ejecuta todos los tests mvn test -Dtest=AppTest # solo tests unitarios mvn test -Dtest=GameEngineIntegrationTest # solo integración mvn surefire-report:report # informe HTML en target/site/

Nota: los tests no necesitan entorno gráfico porque GameEngine y GameLogic son clases Java puras sin dependencia de Swing.

Cómo ampliar el juego sin romper los tests existentes Extensión
  • Más enemigos: añade una List<Rectangle> en GameEngine e itera llamando a GameLogic.hasCollision para cada uno.
  • Dificultad progresiva: incrementa enemyHorizontalSpeed o el rango de velocidades verticales cada N puntos conseguidos.
  • Disparos del jugador: añade una clase Bullet, gestiónala en GameEngine y detecta colisión con el enemigo usando GameLogic.hasCollision.
  • Puntuación máxima: persiste el highscore en un fichero .properties y muéstralo en el HUD junto al score actual.
  • Sonido mejorado: sustituye el Toolkit.beep() por javax.sound.sampled.AudioClip con un fichero WAV.
  • Menú principal: añade un panel de menú y gestiona transiciones con CardLayout en el JFrame.
Dependencias del pom.xml Build
  • JUnit 5 (Jupiter) — motor de tests unitarios y de integración.
  • Maven Surefire Plugin — descubre y ejecuta los tests; genera informes XML en target/surefire-reports/.
  • Maven Exec Plugin — permite lanzar la aplicación con mvn exec:java sin construir un JAR.

Recomendación de arquitectura: mantén App como capa de presentación, GameEngine como único contenedor de estado mutable y GameLogic como librería de funciones puras. Esta separación permite testear la lógica sin levantar ventanas Swing y escalar el juego sin romper los tests existentes.
